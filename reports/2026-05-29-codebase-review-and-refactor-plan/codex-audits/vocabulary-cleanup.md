**Verdicts**

1. **CONFIRM** `*IO` -> `*Transport`, but prefer `SubprocessTransport`, `HttpTransport`, `PyodideTransport`. “Transport” is the standard lower layer term: Thrift has `TTransport`/`TProtocol`; gRPC separates channel/transport from stubs/calls. Current docs already define `Transport` as the byte/message-moving contract ([src/runtime/transport.ts:102](/Users/brettbonner/tywrap/src/runtime/transport.ts:102)), while `ProcessIO` is just a transport implementation ([src/runtime/process-io.ts:165](/Users/brettbonner/tywrap/src/runtime/process-io.ts:165)). Avoid `NodeSubprocessTransport` unless it truly cannot work outside Node.

2. **PUSH BACK** on `ProtocolClient`; use `RpcClient` or `JsonlRpcClient`. The class you want owns request IDs, pending calls, response validation, and dispatch semantics, not “protocol” in the abstract. `BridgeCore` already uses better names: `RpcRequest`/`RpcResponse` ([src/runtime/bridge-core.ts:7](/Users/brettbonner/tywrap/src/runtime/bridge-core.ts:7)) and does correlation with `pending`/`nextId` ([src/runtime/bridge-core.ts:59](/Users/brettbonner/tywrap/src/runtime/bridge-core.ts:59), [src/runtime/bridge-core.ts:107](/Users/brettbonner/tywrap/src/runtime/bridge-core.ts:107)). JSON-RPC calls these Request/Response objects; gRPC would call the public API a stub/client and the connection a channel. Also: current `Transport.send()` and `ProcessIO` already do ID correlation ([src/runtime/process-io.ts:184](/Users/brettbonner/tywrap/src/runtime/process-io.ts:184), [src/runtime/process-io.ts:249](/Users/brettbonner/tywrap/src/runtime/process-io.ts:249)), so the refactor must ensure only one layer owns correlation.

3. **CONFIRM** dropping vibe adjectives, with caveats. `SafeCodec` -> `BridgeCodec` is better than bare `Codec`; `Codec` collides conceptually with `src/utils/codec.ts` value decoding and is too broad ([src/utils/codec.ts:43](/Users/brettbonner/tywrap/src/utils/codec.ts:43)). **Most Important Recommendation**

Define one vocabulary table and enforce it everywhere: **Transport moves bytes/messages; RpcClient owns JSONL RPC IDs and request/response correlation; Bridge is the public Python execution facade; BridgeCodec/ValueCodec encodes values.** Most confusion in this repo comes from letting those four layers share names.
